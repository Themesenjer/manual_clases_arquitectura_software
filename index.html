Semana 1: Atributos de Calidad y El Arte del "Trade-off"

La Arquitectura de Software representa el conjunto de decisiones estratégicas y estructurales que son difíciles de cambiar una vez implementadas. No define aspectos estéticos superficiales, sino el comportamiento y rendimiento del sistema bajo escenarios críticos de presión.

💡 Analogía del Arquitecto: Mientras la programación se asemeja a colocar los ladrillos y el cemento, la arquitectura define la naturaleza del edificio. Un hospital priorizará energía ininterrumpida (Disponibilidad), una discoteca requerirá un piso que soporte altas cargas dinámicas (Escalabilidad), y una vivienda común buscará optimizar el presupuesto (Costo/Mantenibilidad).

El Gran Dilema: El "Trade-off" (Sacrificios)

No existe un sistema de software perfecto que cumpla con ser infinitamente veloz, 100% seguro, gratuito y extremadamente fácil de codificar. La regla de oro dicta que para maximizar un atributo de calidad, inevitablemente se debe mitigar o sacrificar otro.

Requisitos Arquitectónicamente Significativos (ASRs)

  • Disponibilidad (Availability): Mide el porcentaje de tiempo que el sistema permanece operativo y respondiendo (estándar de 99.999%).
  • Escalabilidad (Scalability): Capacidad de absorber incrementos en la carga (Vertical vs Horizontal).
  • Resiliencia: Mecanismos automáticos de recuperación ante fallos imprevistos de infraestructura (Circuit Breaker, Retry).
  • Fiabilidad (Reliability): Probabilidad de operar correctamente bajo métricas MTBF y MTTR.
  • Durabilidad (Persistence & Longevidad): Asegura que los datos no se corrompan o pierdan tras transacciones exitosas, y evalúa cómo envejece el código.

Semana 2: Documentación y Diseño Arquitectónico (ADRs)

La documentación arquitectónica mitiga la pérdida de contexto cuando los equipos cambian en el tiempo. Los ADRs (Architecture Decision Records) capturan el porqué de una decisión técnica crítica.

"Code tells you HOW. Comments tell you WHAT. Documentation tells you WHY."

Sección Práctica: Registro de Decisiones

Ejercicio Resuelto: ADR-001 - Almacenamiento de Videos Resuelto

Problema Evaluado: Elegir la infraestructura óptima para el almacenamiento y entrega masiva de archivos de video para una plataforma de streaming.

Alternativas analizadas: AWS S3 vs CDN Propio vs Filesystem Distribuido en servidores On-Premise.

Solución Seleccionada: AWS S3 integrado con Amazon CloudFront como CDN. Ofrece alta disponibilidad nativa, replicación geográfica automática y reduce sustancialmente el costo operativo inicial (OpEx) frente a mantener almacenamiento en servidores dedicados propios.

Ejercicio Propuesto: Diseño de Cadena de ADRs para Streaming de Video Resuelto por Alumno

Enunciado de la diapositiva: Diseñar la cadena secuencial dependiente de decisiones a partir de ADR-001 (Almacenamiento de video) enfocándose en protocolos de streaming, caché y concurrencia.

ADR-002: Protocolo de Streaming de Video Estatus: Aceptado Contexto: Dado que los videos se almacenan en AWS S3 (ADR-001), necesitamos transmitir el contenido a navegadores web y dispositivos móviles de forma óptima bajo anchos de banda variables, evitando descargar el archivo completo. Decisión: Adoptar HLS (HTTP Live Streaming) de Apple. Justificación: Fragmenta los videos en bloques pequeños de transporte (.ts) y genera archivos de manifiesto (.m3u8). Esto permite el bitrate adaptativo dinámico y posee compatibilidad nativa absoluta en HTML5. Consecuencias: Mayor tiempo de procesamiento inicial para transcodificar videos en S3, pero reduce drásticamente el consumo de ancho de banda en la red.
ADR-003: Estrategia de Caché de Contenido Estatus: Aceptado Contexto: HLS genera miles de peticiones HTTP repetitivas para los mismos fragmentos de video. Golpear a AWS S3 directamente causaría una latencia P99 inaceptable y costos insostenibles. Decisión: Implementar Amazon CloudFront (Edge Caching) acoplado a las políticas de expiración del manifiesto. Justificación: CloudFront replica y almacena los fragmentos de video en más de 400 puntos de presencia globales, reduciendo la latencia de red al mínimo al servir desde el nodo más cercano al estudiante/usuario. Consecuencias: Complejidad menor al invalidar manualmente la caché si un video se edita o reemplaza.
ADR-004: Manejo de Usuarios Concurrentes en Chat en Vivo Estatus: Aceptado Contexto: Para la transmisión didáctica interactiva, los usuarios requieren un canal de chat en tiempo real sincronizado. El HTTP Polling tradicional saturará los servidores web. Decisión: Utilizar conexiones bidireccionales persistentes mediante WebSockets a través de AWS API Gateway. Justificación: Permite comunicación full-duplex de baja latencia con un consumo insignificante de cabeceras de red en comparación con peticiones HTTP repetitivas. Consecuencias: El balanceador de carga debe configurarse para mantener conexiones persistentes largas (stateful).

Los estilos estructurales delimitan las fronteras lógicas y responsabilidades en el código fuente. Analizamos Capas, Microkernel y Pizarra.

Sección Práctica: Aplicación de Estilos Clásicos

Ejercicio Resuelto: Capa de Persistencia Desacoplada Resuelto

Objetivo del ejercicio: Demostrar el desacoplamiento de capas mapeando la alteración de la base de datos sin afectar la interfaz gráfica.

Solución técnica: Separar las consultas mediante un Repositorio. Si la base de datos migra de MySQL a PostgreSQL, el cambio se limita exclusivamente a la capa inferior de almacenamiento, manteniendo la capa de presentación intacta.

Ejercicio Propuesto: Laboratorio en Código - Simulación del Patrón de Pizarra Resuelto por Alumno

Enunciado del PDF: Implementar en código ejecutable una simulación del Patrón de Pizarra (Blackboard) donde múltiples expertos resuelvan un problema no determinista (ej. análisis de telemetría médica o descifrado parcial de datos).

Solución desarrollada en JavaScript puro (Node.js/Navegador):

// 1. La Pizarra (Blackboard) - Almacena el estado y datos del problema
class Blackboard {
    constructor() {
        this.datosMedicos = { presion: 140, ritmoCardiaco: 105, oxigeno: 88 };
        this.diagnosticos = [];
        this.problemaResuelto = false;
    }
}

// 2. Expertos (Knowledge Sources) - Módulos especializados independientes
class ExpertoCardiologia {
    puedeContribuir(blackboard) {
        return blackboard.datosMedicos.ritmoCardiaco > 100 && !blackboard.diagnosticos.includes("Taquicardia Detectada");
    }
    ejecutar(blackboard) {
        blackboard.diagnosticos.push("Taquicardia Detectada");
        console.log("-> [Experto Cardiología]: Ritmo cardíaco elevado. Añadiendo alerta de Taquicardia.");
    }
}

class ExpertoNeumologia {
    puedeContribuir(blackboard) {
        return blackboard.datosMedicos.oxigeno < 90 && !blackboard.diagnosticos.includes("Hipoxia Celular Severa");
    }
    ejecutar(blackboard) {
        blackboard.diagnosticos.push("Hipoxia Celular Severa");
        console.log("-> [Experto Neumología]: Saturación de oxígeno crítica. Añadiendo alerta de Hipoxia.");
    }
}

// 3. El Controlador - Gestiona el bucle de ejecución y prioridades
class Controlador {
    constructor(blackboard) {
        this.blackboard = blackboard;
        this.expertos = [new ExpertoCardiologia(), new ExpertoNeumologia()];
    }

    iniciarBucle() {
        console.log("=== INICIANDO SISTEMA EXPERTO DE PIZARRA ===");
        let cambiosEfectuados = true;

        while (cambiosEfectuados && !this.blackboard.problemaResuelto) {
            cambiosEfectuados = false;
            
            for (let experto of this.expertos) {
                if (experto.puedeContribuir(this.blackboard)) {
                    experto.ejecutar(this.blackboard);
                    cambiosEfectuados = true; // Se altera la pizarra, requiere nueva iteración
                }
            }
            if (!cambiosEfectuados) this.blackboard.problemaResuelto = true;
        }
        console.log("=== ANÁLISIS FINAL DE LA PIZARRA ===");
        console.log("Diagnósticos consolidados:", this.blackboard.diagnosticos);
    }
}

// Ejecución de la simulación didáctica
const pizarraHospital = new Blackboard();
const monitor = new Controlador(pizarraHospital);
monitor.iniciarBucle();

Semana 4: Estilos II — Distribuidos y Observabilidad

Migración de monolitos a servicios fragmentados independientes que se comunican exclusivamente por protocolos de red y exigen observabilidad estricta.

Sección Práctica: Consistencia y Trazabilidad en Microservicios

Ejercicio Resuelto: Composición de API en el Gateway Resuelto

Problema: Un frontend necesita renderizar el perfil del usuario mostrando datos de su cuenta, compras y notificaciones, pero cada módulo reside en un microservicio con su base de datos aislada (Database-per-service).

Solución implementada: El API Gateway recibe una única llamada, ejecuta llamadas concurrentes a los microservicios de cuentas, compras y notificaciones, une las respuestas en un único JSON y lo entrega optimizado al cliente.

Ejercicio Propuesto: Diseño Completo del Patrón SAGA Orquestado Resuelto por Alumno

Enunciado del PDF: Modelar paso a paso una transacción distribuida utilizando el Patrón SAGA Orquestado para un flujo de comercio electrónico (Pedido, Pago e Inventario). Se debe incluir explícitamente el flujo feliz y las transacciones compensatorias en caso de que falle el inventario.

Arquitectura del Flujo Feliz (Paso a Paso):

  1. El cliente solicita comprar. El Servicio de Órdenes crea un registro en estado PENDIENTE_DE_PAGO y delega el control al Orquestador de SAGA.
  2. El Orquestador envía un comando al Servicio de Pagos para procesar el cobro. El servicio cobra con éxito y responde PAGO_CONFIRMADO.
  3. El Orquestador envía un comando al Servicio de Inventario para reservar los productos. El servicio reduce el stock y responde STOCK_RESERVADO.
  4. El Orquestador notifica al Servicio de Órdenes que cambie el estado final a COMPLETADO. El flujo concluye felizmente.

Arquitectura ante Fallo en Inventario (Transacciones Compensatorias ejecutadas por el Orquestador):

Servicio Afectado Transacción Original Ejecutada Transacción Compensatoria (Rollback Lógico)
Servicio de Órdenes Crea orden en estado PENDIENTE Actualiza orden a estado CANCELADA_POR_FALTA_DE_STOCK
Servicio de Pagos Debita el dinero de la tarjeta del cliente ($50) Efectúa un **Reembolso completo** del dinero a la tarjeta
Servicio de Inventario Falla al intentar descontar (Stock insuficiente) No aplica (Emite la señal de error original)

Semana 5: Estilos III — Reactivos, Evolutivos y Adaptabilidad

Ecosistemas reactivos basados en eventos (EDA), segregación de lectura/escritura (CQRS) y control del diseño continuo mediante Fitness Functions en pipelines de CI/CD.

⚡ Regla de Oro CQRS: Las lecturas y las escrituras poseen cargas asimétricas. CQRS permite escalar el servidor de lecturas a 10 instancias si los usuarios consultan mucho, mientras que mantienes el de escrituras estable en una sola instancia.
