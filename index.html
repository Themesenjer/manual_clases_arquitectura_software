Semana 1: Atributos de Calidad y El Arte del "Trade-off"

1.1 ¿Qué es la Arquitectura de Software?

La arquitectura representa el conjunto de decisiones estratégicas fundamentales que dan forma a un sistema informático y que son costosas o difíciles de revertir una vez implementadas. No se enfoca en aspectos estéticos o de interfaz superficiales, sino en las propiedades operativas globales y en cómo se comportará la infraestructura bajo entornos críticos de estrés.

💡 Analogía del Edificio (MSc. Raúl Páez): Imagina que vas a construir un edificio. La programación equivale a colocar de manera física los ladrillos y el cemento. Sin embargo, la Arquitectura consiste en dictaminar racionalmente si esa edificación corresponderá a un hospital, una discoteca o una vivienda familiar.

* Si es un hospital, requerirás sistemas de energía eléctrica ininterrumpida de respaldo (Disponibilidad).
* Si es una discoteca, el piso estructurado debe soportar las cargas dinámicas de mil personas saltando en sincronía (Escalabilidad).
* Si es una vivienda, el foco prioritario será minimizar el presupuesto y facilitar refacciones sencillas (Costo/Mantenibilidad).

1.2 El Gran Dilema: El "Trade-off" (Sacrificios)

En el plano real de la ingeniería de software, un arquitecto debe actuar como un negociador pragmático. No existe el sistema perfecto que sea infinitamente veloz, 100% infranqueable ante ciberataques, completamente gratuito y sumamente sencillo de codificar por programadores junior.

La regla de oro de la disciplina dicta que para elevar o maximizar un atributo de calidad específico, de manera inevitable e intrínseca se debe mitigar, comprometer o sacrificar otro.

1.3 Requisitos Arquitectónicamente Significativos (ASRs)

Los atributos de calidad guían el diseño de las plataformas. Entre los pilares indispensables analizados se encuentran:

  • Disponibilidad (Availability): Mide el porcentaje de tiempo que la plataforma permanece activa, operativa y respondiendo solicitudes correctamente. El estándar de excelencia industrial se calcula en base a la "métrica de los nueves" (ej. un Acuerdo de Nivel de Servicio o SLA de 99.999% representa un límite técnico de caída de apenas 5 minutos anuales). Se obtiene eliminando Puntos Únicos de Fallo (SPOF) acoplando balanceadores de carga y redundancias físicas.
  • Escalabilidad (Scalability): Capacidad inherente del sistema informático para procesar incrementos abruptos de carga transaccional o volumen de datos mediante la inyección estructurada de hardware:
    • Escalabilidad Vertical (Scaling Up): Consiste en repotenciar la máquina existente añadiendo más memoria RAM, CPU o arreglos SSD. Tiene un límite físico estricto e induce costos exponencialmente prohibitivos.
    • Escalabilidad Horizontal (Scaling Out): Consiste en añadir nodos independientes en red distribuyendo el tráfico. Es teóricamente infinita, pero obliga al software a estar diseñado para entornos distribuidos (stateless).
  • Resiliencia: La capacidad del sistema de absorber fallos elásticos de red, hardware o servicios externos y recuperarse de forma automatizada garantizando la continuidad del negocio sin degradaciones catastróficas. Utiliza patrones de contención lógica como el Circuit Breaker (Disyuntor) y políticas de Retry exponenciales.
  • Fiabilidad (Reliability): La probabilidad matemática de que un componente informático ejecute su función esperada sin fallos ni corrupción interna de datos por un intervalo de tiempo delimitado. Se computa usando los indicadores técnicos MTBF (Tiempo Promedio Entre Fallos) y MTTR (Tiempo Promedio de Reparación).
  • Durabilidad (Persistence & Longevidad): En almacenamiento de datos, garantiza que una transacción completada y confirmada persista permanentemente en medios físicos estables (discos no volátiles) incluso si ocurre un colapso eléctrico masivo en el centro de datos un milisegundo después. A nivel estructural, la Durabilidad Arquitectónica o longevidad evalúa la madurez de un código para envejecer con dignidad, permitiendo ser extendido o refactorizado 5 o 10 años después sin necesidad de demolerlo por completo.

Sección Didáctica / Debates de Clase

Ejercicio de Reflexión: "Pensamiento de Arquitecto" - App de Mensajería Analizado

Escenario propuesto por el docente: Analicemos el diseño de una aplicación a gran escala tipo WhatsApp bajo dos ópticas de fallo:

  • Si falla la Resiliencia: El usuario intenta enviar un mensaje de texto, ocurre un microcorte transitorio en el servicio de internet del celular y la aplicación se congela, se cierra abruptamente (crash) o detiene su hilo principal de ejecución. (Genera una experiencia de usuario nefasta).
  • Si falla la Durabilidad: El usuario envía el mensaje, la interfaz gráfica le muestra un check verde de "Enviado con Éxito", pero el servidor central de mensajería se reinicia de imprevisto y el registro se borra de la memoria del clúster antes de persistir. El receptor jamás lee el mensaje. (Provoca pérdida de información crítica e irreversible).

Pregunta para debate en examen: ¿Qué es intrínsecamente peor en un ecosistema transaccional bancario? ¿Que el sistema no responda durante 5 minutos (Falla transitoria de Resiliencia) o que se desvanezca por completo el registro físico de un depósito de $500 ejecutado por un cliente (Falla de Durabilidad)?

Resolución Técnica Obligatoria: En el dominio financiero, una falla de durabilidad destruye la consistencia atómica y legal del balance contable, lo cual es inaceptable y acarrea multas regulatorias severas. Una interrupción de 5 minutos afecta la disponibilidad temporal, la cual puede ser mitigada con colas de mensajería tolerantes a fallos. Por ende, la durabilidad es innegociable frente a la disponibilidad en este dominio.

Semana 2: Documentación y Diseño Arquitectónico (ADRs)

2.1 La Documentación como Herramienta de Comunicación

La documentación en los proyectos de software avanzados no constituye un artefacto burocrático inerte o una pérdida de tiempo. Es, en sentido estricto, infraestructura de conocimiento compartido. Su misión primaria es evitar el colapso del sistema por la pérdida de contexto técnico y la rotación natural del personal en los equipos de ingeniería.

Un documento arquitectónico maduro debe comunicar con claridad meridiana:

  • Decisiones estratégicas de diseño adoptadas y sus justificaciones racionales.
  • Restricciones severas tanto del sistema tecnológico como de las reglas de negocio.
  • Fronteras lógicas, componentes, interfaces expuestas y árboles de dependencias.
  • Riesgos técnicos mitigados y supuestos iniciales de carga de usuarios.

2.2 El Público Objetivo de la Documentación

Para que el conocimiento sea líquido, debe estructurarse para diferentes perfiles (stakeholders):

  • Desarrolladores: Guía exacta para codificación limpia, patrones de diseño y mantenimiento diario.
  • Arquitectos: Hoja de ruta para asegurar la coherencia del ecosistema ante cambios y evoluciones futuras.
  • QA / Testers: Definición clara de los criterios de aceptación de calidad y coberturas críticas a probar.
  • Product Owners / Negocio: Delimitación exacta del alcance, restricciones operativas y viabilidad comercial.
  • Futuros Equipos: Onboarding acelerado y continuidad técnica sin fricciones.
"Code tells you HOW. Comments tell you WHAT. Documentation tells you WHY."

2.3 Architecture Decision Records (ADRs)

Un ADR es un registro conciso que captura una decisión de diseño arquitectónico de alto impacto, documentando meticulosamente el contexto habilitante, las razones técnicas subyacentes, las opciones descartadas y los compromisos asumidos a futuro.

2.4 Anatomía Estricta de un ADR Estructurado

  1. Título: Identificador numérico secuencial acompañado de la declaración de la acción (Ej: ADR-001: Migración de Arquitectura Monolítica a Microservicios).
  2. Estatus: Ciclo de vida dinámico del documento (Propuesto, Aceptado, Rechazado, o Superado por un ADR posterior).
  3. Contexto: Redacción de las fuerzas del negocio, limitaciones presupuestarias, requerimientos técnicos, estado actual del sistema y fricciones observadas que impulsan la necesidad de decidir.
  4. Decisión: Enunciado explícito, directo y prescriptivo del curso tecnológico o estructural que el equipo adoptará de forma obligatoria.
  5. Consecuencias: Análisis detallado y honesto de los compromisos. Debe reflejar tanto los impactos positivos tangibles como los impactos negativos (la deuda técnica asumida o los trade-offs sacrificados) junto a su respectivo plan de mitigación.

Sección Práctica / Taller de ADRs

Ejercicio Resuelto en Clase: ADR-001 - Infraestructura de Almacenamiento de Video Clase

Problema Planteado: Determinar el mecanismo idóneo para el almacenamiento y despacho de miles de archivos multimedia pesados para una plataforma educativa global.

Alternativas Evaluadas: AWS S3 vs CDN Local Dedicado vs Filesystem Distribuido en hardware On-Premise.

Decisión Tomada: Adoptar AWS S3 con Amazon CloudFront. Provee almacenamiento virtualmente ilimitado e inmutable, alta disponibilidad nativa sin sobrecarga de administración de discos y reduce el gasto de capital (CapEx) a costos netamente operativos (OpEx).

Ejercicio Propuesto: Diseño de una Cadena Secuencial de ADRs para Streaming Resuelto por Alumno

Enunciado de la Diapositiva: Tomando como raíz el ADR-001 de almacenamiento en S3, diseñe la cadena lógica secuencial de decisiones arquitectónicas subsecuentes que hereden sus restricciones.

ADR-002: Protocolo de Streaming de Video Estatus: Aceptado Contexto: Al estar los archivos alojados de forma estática en AWS S3 (ADR-001), no podemos obligar a los estudiantes a descargar archivos íntegros de 500MB en redes móviles de baja calidad antes de reproducir las clases. Decisión: Implementar el protocolo HLS (HTTP Live Streaming) de Apple. Justificación: HLS segmenta el flujo multimedia en fragmentos discretos de transporte (.ts) indexados por archivos de texto manifiesto (.m3u8). Esto viabiliza el Bitrate Adaptativo: la calidad del video mutará dinámicamente según el ancho de banda del alumno. Consecuencias: Requiere una tarea asíncrona de transcodificación (mediante AWS Elemental MediaConvert) cada vez que un docente suba un video, incrementando los costos iniciales de procesamiento, pero garantizando fluidez móvil absoluta.
ADR-003: Arquitectura y Distribución de Caché Perimetral Estatus: Aceptado Contexto: Servir los bloques estáticos .ts y manifiestos .m3u8 directamente desde la región de origen de AWS S3 para miles de alumnos concurrentes provocaría latencias elevadas fuera del continente e incrementaría exponencialmente los costos de salida de datos (Egress fees). Decisión: Desplegar **Amazon CloudFront** como Red de Distribución de Contenidos (CDN) interconectado con S3. Justificación: CloudFront interceptará las peticiones y cacheará los bloques de video en las ubicaciones de borde (Edge Locations) cercanas a los nodos de red de los estudiantes, entregando latencias inferiores a 50ms en la descarga de fragmentos. Consecuencias: Obliga a configurar cabeceras HTTP de control de caché (Cache-Control) estrictas y a implementar un mecanismo automático de invalidación si el material educativo es actualizado de emergencia.
ADR-004: Gestión de Alta Concurrencia en Chat en Vivo Interactivos Estatus: Aceptado Contexto: En clases transmitidas por streaming, los estudiantes requieren un chat de interacción masiva en tiempo real. Utilizar arquitecturas REST tradicionales con HTTP Polling constante saturaría los hilos de los servidores web con peticiones redundantes. Decisión: Utilizar conexiones persistentes fullduplex mediante el protocolo **WebSockets** gestionado en AWS API Gateway. Justificación: WebSockets reduce la sobrecarga de transporte al eliminar las pesadas cabeceras HTTP de cada mensaje, manteniendo una única conexión TCP abierta para enviar y recibir datos instantáneamente. Consecuencias: Desplaza la arquitectura web de un modelo puramente sin estado (stateless) a uno con estado persistente (stateful), obligando a rastrear las sesiones activas en una base de datos en memoria ultrarrápida como Redis.

Semana 3: Estilos Arquitectónicos I — Estructurales y Clásicos

3.1 La Fundación Estructural

Los estilos arquitectónicos estructurales proveen patrones predecibles de organización interna para los sistemas de software. Su meta principal es mitigar la entropía y la complejidad del código delimitando de forma estricta las fronteras operativas y los niveles de abstracción lógicos.

3.2 Arquitectura en Capas (N-Tier)

Es el estilo fundacional más extendido. Consiste en particionar el sistema en divisiones horizontales donde cada nivel posee un rol y responsabilidad única (comúnmente: Presentación, Lógica de Negocio, Datos).

  • Aislamiento de Capas: Un cambio técnico en la base de datos subyacente no debería derramarse ni exigir modificaciones en la capa superior de interfaces de usuario.
  • Seguridad Integrada: Cada nivel actúa de forma nativa como una frontera de validación estricta que sanitiza las peticiones entrantes.
  • Riesgo Crítico - Sinkhole Pattern: Antipatrón severo en el cual las solicitudes transitan a través de múltiples capas del software de forma secuencial sin ejecutar ningún procesamiento de valor o lógica de negocio real, actuando como simples pasamanos de datos y degradando la latencia del microsegundo.

3.3 Arquitectura de Microkernel (Plug-ins)

Este estilo desacopla de forma drástica las operaciones básicas del sistema de aquellas funcionalidades complementarias o altamente volátiles.

  • Core System: Alberga la lógica algorítmica mínima indispensable para que la aplicación exista y se mantenga en ejecución. Es rígido y altamente optimizado.
  • Plug-ins: Componentes de software independientes y aislados que inyectan reglas de negocio especializadas o conectores adicionales al Core sin alterarlo internamente.
  • Ámbito de Uso: Es el estándar absoluto para construir Entornos de Desarrollo Integrados (ej. VS Code, Eclipse) y Navegadores Web. Su reto operativo es el infierno de dependencias al gestionar versiones dispares de plug-ins de terceros.

3.4 Patrón de Pizarra (Blackboard Pattern)

Estilo arquitectónico diseñado específicamente para resolver problemas complejos y de naturaleza **no determinista**, donde no se dispone de un algoritmo matemático secuencial cerrado aplicable de forma unívoca (ej: Sistemas expertos de diagnóstico médico, Inteligencia Artificial avanzada, Reconocimiento lingüístico o de voz).

  • La Pizarra (Blackboard): Un repositorio de memoria global y centralizado que contiene los datos del problema y el estado del entorno.
  • Los Expertos (Knowledge Sources): Módulos funcionales totalmente aislados y autónomos. Cada uno es especialista en resolver una subparte muy concreta del problema general. No se comunican entre sí; su única interacción es leer y escribir sobre la pizarra.
  • El Controlador: Un bucle de evaluación que coordina la pizarra, evalúa el estado actual y determina heurísticamente qué experto posee la mayor probabilidad de éxito para activarse en el ciclo de reloj actual. Su gran vulnerabilidad es que se vuelve extremadamente complejo escribir pruebas unitarias deterministas automatizadas.

Laboratorio Programado: Simulación Funcional del Patrón de Pizarra Resuelto por Alumno

Reto del PDF de la materia: Desarrollar en código nativo limpio una simulación ejecutable del patrón Blackboard donde un controlador coordine expertos de salud sobre una memoria centralizada.

Código JavaScript Semántico y Académico:

// 1. ESTRUCTURA CENTRAL DE LA PIZARRA (BLACKBOARD)
class Blackboard {
    constructor() {
        // Estado inicial de telemetría de un paciente en emergencias
        this.telemetriaPaciente = { presionArterial: 145, pulsoCardiaco: 110, saturacionOxigeno: 86 };
        this.alertasDiagnosticas = [];
        this.analisisFinalizado = false;
    }
}

// 2. ENTORNO DE EXPERTOS AUTÓNOMOS (KNOWLEDGE SOURCES)
class ExpertoCardiologia {
    puedeContribuir(blackboard) {
        // Se activa si hay taquicardia latente y la alerta no ha sido escrita
        return blackboard.telemetriaPaciente.pulsoCardiaco > 100 && 
               !blackboard.alertasDiagnosticas.includes("ALERTA CRÍTICA: Taquicardia Detectada");
    }
    ejecutar(blackboard) {
        blackboard.alertasDiagnosticas.push("ALERTA CRÍTICA: Taquicardia Detectada");
        console.log("-> [Experto Cardiología]: Pulso crítico evaluado. Escribiendo diagnóstico en la Pizarra.");
    }
}

class ExpertoNeumologia {
    puedeContribuir(blackboard) {
        // Se activa ante hipoxia severa no registrada en la pizarra
        return blackboard.telemetriaPaciente.saturacionOxigeno < 90 && 
               !blackboard.alertasDiagnosticas.includes("ALERTA CRÍTICA: Hipoxia Celular Aguda");
    }
    ejecutar(blackboard) {
        blackboard.alertasDiagnosticas.push("ALERTA CRÍTICA: Hipoxia Celular Aguda");
        console.log("-> [Experto Neumología]: Oxigenación deficiente. Escribiendo diagnóstico en la Pizarra.");
    }
}

// 3. COMPONENTE DE CONTROL (DESPACHADOR HEURÍSTICO)
class ControladorCentral {
    constructor(blackboard) {
        this.blackboard = blackboard;
        this.catalogoExpertos = [new ExpertoCardiologia(), new ExpertoNeumologia()];
    }

    iniciarCicloProcesamiento() {
        console.log("=== INICIANDO MOTOR DE EVALUACIÓN BLACKBOARD ===");
        let huboCambiosEnCiclo = true;

        // El ciclo corre de manera optimista mientras los expertos sigan aportando conocimiento
        while (huboCambiosEnCiclo && !this.blackboard.analisisFinalizado) {
            huboCambiosEnCiclo = false;

            for (let experto of this.catalogoExpertos) {
                if (experto.puedeContribuir(this.blackboard)) {
                    experto.ejecutar(this.blackboard);
                    huboCambiosEnCiclo = true; // Forzar nueva iteración para evaluar re-impactos
                }
            }
            if (!huboCambiosEnCiclo) {
                this.blackboard.analisisFinalizado = true;
            }
        }

        console.log("=== EVALUACIÓN CONSOLIDADA EN LA PIZARRA ===");
        console.log("Historial de Diagnósticos:", this.blackboard.alertasDiagnosticas);
    }
}

// INSTANCIACIÓN DIDÁCTICA PARA EXAMEN
const casoClinico = new Blackboard();
const evaluador = new ControladorCentral(casoClinico);
evaluador.IniciarCicloProcesamiento();

Semana 4: Estilos II — Distribuidos y Observabilidad

4.1 El Cambio de Paradigma

Iniciamos la transición drástica desde arquitecturas monolíticas de proceso único tradicionales (donde todos los módulos del sistema conviven compartiendo memoria física en un único servidor) hacia **ecosistemas fragmentados distribuidos** que se ven obligados a interactuar cruzando la topología de la red mediante protocolos de transporte.

4.2 Anatomía de los Microservicios

  • Descomposición Funcional (Bounded Context): Cada unidad de despliegue se circunscribe exclusivamente a un dominio acotado del negocio. Esta estricta separación reduce sustancialmente el radio de explosión ante incidentes críticos de producción: si el microservicio secundario de "Sugerencias o Comentarios" colapsa, el core del flujo transaccional de "Ventas" sigue operando perfectamente en línea.
  • Despliegue Independiente: Otorga autonomía organizacional e industrial completa. El equipo de "Pasarelas de Pago" puede empaquetar, compilar e instalar parches en producción múltiples veces al día de forma ágil sin necesidad de solicitar permisos ni congelar las operaciones de los ingenieros del equipo de "Inventario o Catálogo".

4.3 Las Reglas de Oro de los Datos Distribuidos

  • Database-per-service: Queda estrictamente prohibido que múltiples microservicios compartan accesos directos a tablas en la misma base de datos relacional. Eso introduce un acoplamiento oculto por datos destructivo. Cada servicio debe ser dueño absoluto de su persistencia.
  • API Composition: Cuando las vistas del usuario demandan datos heterogéneos dispersos en la red, el componente centralizado **API Gateway** asume la responsabilidad de disparar llamadas paralelas concurrentes, orquestar y compactar las respuestas en una carga útil limpia (JSON unificado) optimizada para el cliente.
  • Idempotencia Obligatoria: Atributo técnico no negociable en redes distribuidas. Asegura y garantiza que procesar la misma transacción exacta o mensaje duplicado dos o más veces en un servidor de destino no alterará de ninguna manera el balance de estado final del sistema ni duplicará cobros monetarios erróneos.

4.4 Consistencia de Datos en Red: 2PC vs. Patrón SAGA

Métrica Operativa Two-Phase Commit (2PC) Patrón SAGA (Arquitectura Distribuida)
Modelo de Consistencia Consistencia Fuerte y Atómica (Inmediata) Consistencia Eventual (Diferida en el tiempo)
Bloqueo de Infraestructura Sí (Bloquea registros de bases de datos de extremo a extremo) No (Es optimista, no congela recursos globales)
Nivel de Escalabilidad Baja (Produce cuellos de botella severos en red) Alta (Flujos puramente asíncronos desacoplados)
Estrategia frente a Errores Rollback Transaccional Automático nativo Ejecución secuencial de **Transacciones Compensatorias**

4.5 La Nueva Era de la Observabilidad

La observabilidad trasciende al monitoreo reactivo clásico. Se define formalmente como la capacidad matemática y de ingeniería para deducir e inferir de forma precisa los estados de salud internos e incidencias anómalas de un ecosistema distribuido analizando únicamente sus salidas externas. Se apoya en **Los Tres Pilares de la Observabilidad**:

  • Logs (Registros Históricos): Líneas de texto inmutables estampadas con marcas de tiempo atómicas que describen con precisión el qué y el cuándo de un evento.
  • Métricas (Datos Agregados): Mediciones numéricas continuas utilizadas para auditar la telemetría del hardware y software (ej. porcentajes de saturación de CPU, consumo de memoria interna, tasa de errores HTTP 5xx por minuto).
  • Traces (Trazabilidad Distribuida): Grafos de flujo que mapean el viaje completo y los hilos de ejecución de una solicitud HTTP a través de toda la malla compleja de microservicios. Utiliza IDs correlacionales para rastrear cuellos de botella de red y latencias anómalas en el percentil **P99** (garantizando que incluso los usuarios más lentos obtengan respuestas en tiempos aceptables sub-200ms).

Reto Propuesto: Modelado Completo del Patrón SAGA Orquestado con Falla de Stock Resuelto por Alumno

Requerimiento del PDF de la Materia: Modelar paso a paso el ciclo de vida transaccional de una compra en línea utilizando el Patrón SAGA Orquestado que involucre el Servicio de Órdenes, Servicio de Pagos y Servicio de Inventario. Debe detallar explícitamente el flujo de excepción y reversión lógica si el inventario detecta stock insuficiente.

Flujo Operativo de Excepción ante Fallas:

  1. El cliente envía la solicitud de compra. El **Servicio de Órdenes** crea un registro inicial en su base de datos local en estado ORDEN_PENDIENTE y notifica al componente central **Orquestador de SAGA**.
  2. El Orquestador toma el control y despacha un comando imperativo directo al **Servicio de Pagos** ordenando ejecutar el cobro. El Servicio de Pagos debita exitosamente los $50 de la cuenta del cliente y responde al Orquestador con el estatus TRANSACCION_DE_PAGO_EXITOSA.
  3. El Orquestador recibe la confirmación financiera e inmediatamente despacha un comando al **Servicio de Inventario** solicitando reservar el stock físico de los productos comprados.
  4. El **Servicio de Inventario** ejecuta la consulta de actualización concurrente en su base de datos local y detecta que otra sesión agotó las existencias (Stock Insuficiente). El servicio rechaza la operación y responde al Orquestador con el código de error ERROR_INVENTARIO_AGOTADO.
  5. Al recibir la señal de colapso, el Orquestador activa de inmediato el bucle de mitigación disparando de manera secuencial las **Transacciones Compensatorias (Rollback Lógico)** para restaurar la consistencia eventual del ecosistema.

Semana 5: Estilos III — Arquitecturas Reactivas, Evolutivas y Adaptabilidad

5.1 Arquitectura Orientada a Eventos (EDA)

Representa la evolución hacia sistemas reactivos asíncronos en tiempo real. En lugar de forzar llamadas acopladas HTTP directas entre servicios, los componentes emiten notificaciones inmutables denominadas **Eventos de Dominio** que indican que algo relevante ha sucedido en el negocio.

Se implementa bajo el patrón de arquitectura **Publicador / Suscriptor (Pub-Sub)**:

  • Productores: Componentes de software encargados de instanciar, empaquetar y emitir el evento en red tras completar una acción de negocio.
  • Event Brokers: Infraestructura masiva de transporte y persistencia de mensajería asíncrona encargada de enrutar y ordenar los flujos informáticos (ej. Apache Kafka, RabbitMQ).
  • Consumidores: Servicios independientes que se suscriben voluntariamente a tópicos de su competencia e interés, reaccionando de manera asíncrona ante la recepción de los mensajes. Permite el máximo desacoplamiento espacial y temporal de la red de software.

5.2 Patrón CQRS (Command Query Responsibility Segregation)

Establece la separación radical e intransigente de los modelos informáticos dedicados a la mutación de datos de aquellos dedicados a las consultas de información.

  • Commands (Comandos): Peticiones e hilos de ejecución orientados exclusivamente a alterar el estado del sistema (Escrituras, Actualizaciones, Eliminaciones). Atraviesan capas complejas de lógica de dominio, validaciones de negocio e inmutabilidades en el Write Model.
  • Queries (Consultas): Peticiones de solo lectura orientadas a retornar Data Transfer Objects (DTOs) optimizados para el consumidor en el Read Model. No ejecutan lógica de negocio. Utilizan almacenes de datos desnormalizados de lectura ultrarrápida.
⚡ Justificación de CQRS (MSc. Raúl Páez): En el mundo real, las lecturas suelen ser 100 veces más frecuentes que las escrituras en un software transaccional masivo. CQRS permite escalar de manera asimétrica la infraestructura: puedes tener 15 servidores dedicados a responder consultas rápidas de forma desnormalizada y mantener una sola instancia robusta procesando las escrituras validadas.

5.3 Arquitectura Evolutiva

Reconoce que el software comercial no es un producto estático, sino un organismo dinámico sujeto a modificaciones continuas del mercado. Una arquitectura evolutiva se diseña de forma consciente para soportar cambios e incrementos guiados a través de múltiples dimensiones operativas a lo largo de su ciclo de vida, erradicando por completo la rigidez y fragilidad del diseño inicial.

5.4 Funciones de Aptitud (Fitness Functions)

Son los **guardianes automatizados del diseño arquitectónico**. Se definen matemáticamente como funciones y métricas objetivas programadas directamente dentro de los flujos automáticos de los pipelines de Integración Continua y Despliegue Continuo (CI/CD) ($f(a) \rightarrow [0,1]$).

Su misión es evaluar de forma continua en cada commit qué tan alineada está la base de código del proyecto con los objetivos de calidad preestablecidos, bloqueando de manera automatizada el paso a entornos productivos si se detectan desvíos severos en:

  • Rendimiento y latencias máximas tolerables (Performance).
  • Brechas y vulnerabilidades críticas de seguridad (Security).
  • Límites permitidos de deuda técnica y acoplamiento cíclico de paquetes (Mantenibilidad).
  • Métricas de cobertura y legalidad de software.

Semana 6: Patrones de Diseño I — Catálogo Creacional y Estructural

6.1 Conceptos Fundamentales de la Programación Orientada a Objetos (POO)

Antes de sumergirse en el catálogo de soluciones de la Gang of Four (GoF), es imperativo dominar los fundamentos que sostienen el diseño robusto de software:

  • Abstracción en POO: Es el proceso mental y técnico mediante el cual aislamos las características esenciales de una entidad del mundo real, descartando los detalles superfluos o complejos de implementación. Permite modelar clases enfocadas únicamente en el qué hace un objeto antes de preocuparse por el cómo lo hace.
  • El Constructor en POO: Es un método especial de una clase que se ejecuta de manera automática al instanciar un objeto. Su propósito arquitectónico es garantizar el correcto ciclo de vida del objeto, reservando espacio en la memoria RAM e inicializando de forma segura sus atributos obligatorios con valores estables o parámetros iniciales.
  • Instanciar: Significa materializar un objeto específico y concreto en la memoria del computador a partir de un molde genérico predefinido llamado clase.
  • Familias de Objetos en POO: Conjunto de componentes de software estrechamente correlacionados o dependientes entre sí que comparten un tema conceptual, estético o funcional común y están estrictamente diseñados para operar juntos de manera coherente en tiempo de ejecución (ej: botones, barras de scroll y ventanas pertenecientes exclusivamente al tema gráfico "Dark Mode").
💡 Analogía de la Instanciación y Abstracción: La clase equivale a los planos arquitectónicos de un vehículo (Abstracción de un automóvil). El **constructor** es la línea de ensamblaje en la fábrica que toma los materiales, y la **instanciación** es el acto exacto en el que sale un auto físico con un número de chasis único a la calle (el Objeto).

6.2 Categorías de la Gang of Four (GoF)

Los 23 patrones clásicos de la GoF se clasifican según su propósito e intención técnica en tres grandes familias:

  1. Creacionales: Abstraen y flexibilizan los mecanismos de creación e instanciación de objetos. Evitan que el sistema dependa del operador directo new o de configuraciones rígidas de constructores.
  2. Estructurales: Definen la forma idónea en que se componen, entrelazan e interactúan las clases y los objetos para conformar estructuras mucho más grandes y complejas de manera flexible y escalable, favoreciendo la composición sobre la herencia.
  3. Comportamiento: Gestionan los algoritmos, la asignación de responsabilidades y los flujos de comunicación dinámica entre los objetos interconectados.

6.3 Patrones Creacionales en Detalle

Abstraen el proceso de instanciación para desacoplar el código de los detalles de inicialización:

  • Singleton: Garantiza de forma estricta que una clase posea una única e inequívoca instancia viva en todo el ciclo de vida del sistema, proveyendo un punto de acceso global y centralizado a dicho recurso. Ejemplo: Un administrador único de configuraciones de la app o un Pool centralizado de Conexiones a Base de Datos.
  • Factory Method: Define una interfaz abstracta para la creación de un objeto, pero delega la responsabilidad a las subclases para que decidan la clase exacta a instanciar en tiempo de ejecución. Ejemplo: Un sistema logístico donde una fábrica despacha transporte terrestre (Camion) o marítimo (Barco) dependiendo del destino.
  • Abstract Factory: Proporciona una interfaz para producir familias completas de objetos relacionados o dependientes entre sí sin necesidad de acoplarse o especificar sus clases concretas directas. Ejemplo: Un kit de herramientas UI que genera interfaces nativas consistentes para Windows (BotonWin, CheckboxWin) o Mac (BotonMac, CheckboxMac).
  • Builder: Descompone y separa el proceso de construcción de un objeto complejo de su representación final, permitiendo que el mismo proceso de construcción paso a paso sea capaz de moldear múltiples variaciones distintas. Ejemplo: Un constructor de consultas complejas de base de datos (QueryBuilder) agregando de forma condicional select(), where() y join().
  • Prototype: Permite la clonación o copia exacta de objetos preexistentes duplicando directamente su estado en memoria RAM, sin que el código de cliente tenga que depender de las clases específicas ni de costosos constructores iniciales. Ejemplo: Duplicar miles de monstruos o enemigos en un videojuego que comparten texturas y estadísticas idénticas, variando solo su posición en pantalla.

6.4 Análisis de Código Avanzado: Singleton de Alto Impacto (Diapositiva 5 Desarrollada)

A continuación se expande el ejemplo elemental de la presentación hacia un Singleton robusto y seguro para entornos de alta concurrencia multihilo (Thread-Safe), esencial en la arquitectura corporativa moderna:

import threading

class DatabaseConnectionPool:
    _instance = None
    _lock = threading.Lock()  # Lock de exclusión mutua para entornos multihilo

    def __new__(cls):
        # Implementación del patrón de doble verificación (Double-Checked Locking)
        if cls._instance is None:
            with cls._lock:
                if cls._instance is None:
                    cls._instance = super().__new__(cls)
                    cls._instance._inicializar_pool()
        return cls._instance

    def _inicializar_pool(self):
        self.conexion_string = "postgresql://db_user:secret@localhost:5432/produccion"
        self.max_conexiones = 20
        self.conexiones_activas = []
        print(" [CONEXIÓN ÚNICA] Pool de conexiones inicializado de forma segura y única en memoria.")

    def obtener_conexion(self):
        return f"Conexión activa despachada bajo {self.conexion_string}"

# Demostración en tiempo de ejecución
pool_a = DatabaseConnectionPool()
pool_b = DatabaseConnectionPool()

print(f"¿Ambas variables apuntan a la misma instancia en memoria?: {pool_a is pool_b}")

6.5 Patrones Estructurales en Detalle

Se enfocan en cómo las clases y los objetos se entrelazan armónicamente para dar soporte a sistemas expansivos:

  • Adapter: Actúa como un traductor estructural que permite que dos interfaces inicialmente incompatibles colaboren y trabajen juntas de forma transparente, envolviendo la interfaz antigua o externa para adecuarla a la firma esperada por el cliente. Ejemplo real fuera de SO: Un lector físico de tarjetas SD adaptado y conectado a un puerto USB de una computadora actual, o un convertidor de corriente de enchufe europeo a toma americana.
  • Bridge: Desacopla y separa radicalmente una abstracción de su implementación práctica, posibilitando que ambas dimensiones de diseño evolucionen, varíen y crezcan de forma totalmente independiente. Ejemplo conceptual: El control remoto universal (Abstracción de control) frente a las televisiones concretas de marcas como Sony o Samsung (Implementación). Se pueden añadir nuevos botones al control sin alterar la electrónica interna de los televisores.
  • Composite: Estructura y compone objetos en complejas organizaciones arbóreas para representar jerarquías de tipo parte-todo, logrando que el cliente trate exactamente por igual a los elementos individuales simples y a las composiciones colectivas ramificadas. Ejemplo en POO: Un sistema de facturación de menús donde un ítem puede ser un producto individual (Hamburguesa) o un combo de productos (ComboFamiliar que agrupa papas, gaseosas y hamburguesas). Ambos heredan de una interfaz común Vendible y exponen el método calcular_precio() de manera idéntica.
  • Decorator: Inyecta responsabilidades, lógicas funcionales o comportamientos adicionales a un objeto específico de forma completamente dinámica en caliente, sin alterar en absoluto su estructura interna original y evitando el crecimiento desmesurado de la jerarquía por herencia rígida. Ejemplo conceptual: La preparación de un café base. Tienes el CafeSolo y dinámicamente le añades decoradores como ConLeche, ConChocolate o ConAzucar, acumulando costos y propiedades de forma líquida.
  • Facade (Fachada): Proporciona una interfaz unificada, extremadamente simplificada y de alto nivel que sirve de compuerta única de acceso frente a un subsistema complejo de clases ocultando toda la fricción técnica y llamadas engorrosas del backend. Ejemplo conceptual: El botón "Comprar con 1 clic" en una tienda e-commerce. Oculta el proceso en cadena de verificar inventario físico, procesar pago con banco externo, emitir factura legal en PDF y coordinar con la empresa transportista.
  • Flyweight (Peso Ligero): Minimiza el consumo crítico de memoria RAM compartiendo de manera eficiente y centralizada fragmentos de estados comunes e intrínsecos entre una cantidad masiva de objetos de naturaleza similar. Ejemplo conceptual: Un simulador de bosque en un videojuego con 100,000 árboles. Los datos pesados comunes (modelo 3D de las hojas, texturas de la corteza) se almacenan en una única instancia Flyweight, mientras que los datos extrínsecos ligeros (posición X, Y, Z de cada árbol y nivel de daño) se almacenan por separado.

6.6 Solución de Ingeniería: Resolución del Problema de Integración (Diapositiva 9)

La presentación plantea un problema clásico de arquitectura: la necesidad de integrar una API externa de pagos internacionales que expone el método estricto pay_usd(), mientras que nuestra arquitectura local ya consolidada exige de manera obligatoria interactuar a través de la firma procesar_cobro().

A continuación se detalla la simulación y resolución exacta mediante el patrón **Adapter**:

# 1. Componente de Terceros (Incompatible e Inmodificable)
class GatewayPagoExternoUSD:
    def pay_usd(self, amount_usd: float):
        print(f" -> [API Terceros] Transacción autorizada exitosamente por ${amount_usd} USD.")

# 2. Interfaz Esperada por Nuestra Arquitectura Local
class TARGET_ProcesadorPagoLocal:
    def procesar_cobro(self, monto_hitos: float):
        raise NotImplementedError("Se debe implementar la interfaz local")

# 3. El ADAPTER: Encapsula la API vieja/externa y expone la firma nueva
class AdapterPagoInternacional(TARGET_ProcesadorPagoLocal):
    def __init__(self, api_externa: GatewayPagoExternoUSD):
        self._api_externa = api_externa

    def procesar_cobro(self, monto_hitos: float):
        # El Adapter asume la lógica de traducción interna de la petición
        print(f"[ADAPTER] Convirtiendo petición local hacia formato internacional...")
        # Simulación de mapeo o conversión de negocio (ej. conversión de moneda)
        self._api_externa.pay_usd(monto_hitos)

# 4. Código de Cliente (Arquitectura Local del IST Yavirac)
if __name__ == "__main__":
    print("--- SISTEMA DE COBROS INSTITUCIONAL ---")
    
    # El cliente no sabe que por detrás hay una API en USD incompatible
    api_externa_usd = GatewayPagoExternoUSD()
    procesador_con_adapter = AdapterPagoInternacional(api_externa_usd)
    
    # La llamada se hace con la firma local que espera nuestra arquitectura
    procesador_con_adapter.procesar_cobro(150.00)

Semana 7: Patrones de Diseño II — Algoritmos y Asignación de Responsabilidades

7.1 Conceptos Fundamentales de los Patrones de Comportamiento

A diferencia de los creacionales (instanciación) y estructurales (composición), los Patrones de Comportamiento se concentran específicamente en los algoritmos, los flujos de ejecución y la distribución de responsabilidades críticas entre objetos interconectados.

  • Infructructura de Comunicación: No solo describen objetos o clases de forma aislada, sino el diseño de los canales abstractos y dinámicos a través de los cuales se comunican en tiempo de ejecución.
  • Desacoplamiento Operacional: Su objetivo fundamental es liberar el flujo de ejecución de conexiones rígidas o cableadas (hardcoded).
  • Foco de Abstracción: Encapsulan procesos y comportamientos variables del negocio detrás de abstracciones estables, permitiendo cambiar drásticamente cómo responde el sistema ante un evento sin modificar su código núcleo.

7.2 Catálogo de Patrones de Comportamiento con Ejemplos y Explicaciones

A continuación se examinan y desarrollan los patrones clave de la GoF indicados para la arquitectura táctica moderna:

A. Chain of Responsibility (Cadena de Responsabilidad)

Teoría: Permite pasar una petición de negocio a lo largo de una cadena secuencial de manejadores independientes. Cada manejador decide si procesa internamente la solicitud o si la delega al siguiente eslabón en la fila.

# Ejemplo: Cadena de aprobación de presupuestos institucionales
class ManejadorAprobacion:
    def __init__(self, sucesor=None):
        self._sucesor = sucesor

    def procesar_solicitud(self, monto: float):
        if self._sucesor:
            self._sucesor.procesar_solicitud(monto)

class Coordinador(ManejadorAprobacion):
    def procesar_solicitud(self, monto: float):
        if monto <= 500:
            print(f"[COORDINADOR] Aprobado automáticamente monto menor a $500.")
        else:
            print(f"[COORDINADOR] Excede mi límite. Pasando al Decano...")
            super().procesar_solicitud(monto)

class Decano(ManejadorAprobacion):
    def procesar_solicitud(self, monto: float):
        if monto <= 5000:
            print(f"[DECANO] Aprobado formalmente presupuesto de ${monto}.")
        else:
            print(f"[DECANO] Monto crítico. Pasando a Rectoría...")
            super().procesar_solicitud(monto)

class Rector(ManejadorAprobacion):
    def procesar_solicitud(self, monto: float):
        print(f"[RECTOR] Revisión exhaustiva y aprobación final para ${monto}.")

# Uso
cadena = Coordinador(Decano(Rector()))
cadena.procesar_solicitud(3500)

Explicación: El cliente realiza una sola llamada al Coordinador sin saber quién resolverá la petición en última instancia. Si el monto es alto ($3500), el objeto delega dinámicamente al Decano evitando condicionales de tipo if/else anidados y gigantescos en la capa de negocio.

B. Command (Comando)

Teoría: Transforma una solicitud o acción operativa en un objeto de negocio independiente que contiene toda la información necesaria para ejecutarla. Esto permite parametrizar operaciones, encolarlas y soportar operaciones de deshacer (undo).

# Ejemplo: Operaciones del sistema de base de datos transaccional
class Comando:
    def ejecutar(self): pass
    def deshacer(self): pass

class GuardarUsuarioComando(Comando):
    def __init__(self, bd, usuario_id, datos):
        self.bd = bd
        self.usuario_id = usuario_id
        self.datos = datos

    def ejecutar(self):
        self.bd.insertar(self.usuario_id, self.datos)

    def deshacer(self):
        self.bd.eliminar(self.usuario_id)

class InvocadorHistorial:
    def __init__(self):
        self._historial = []

    def ejecutar_comando(self, comando: Comando):
        comando.ejecutar()
        self._historial.append(comando)

    def deshacer_ultimo(self):
        if self._historial:
            comando = self._historial.pop()
            comando.deshacer()

Explicación: Al encapsular la inserción como un objeto Comando, el sistema gana la capacidad de almacenar acciones en un array (historial), permitiendo volver atrás en el tiempo (Rollback) de forma limpia llamando a deshacer() sin acoplar la UI a la persistencia.

C. Strategy (Estrategia)

Teoría: Define una familia de algoritmos intercambiables, encapsula cada uno de ellos en una clase independiente y logra que operen de forma transparente para el cliente, pudiendo cambiar de algoritmo en tiempo de ejecución de acuerdo a la necesidad de negocio.

# Ejemplo: Pasarelas de cobro dinámicas en el IST Yavirac
class EstrategiaPago:
    def pagar(self, monto: float): pass

class PagoTarjeta(EstrategiaPago):
    def pagar(self, monto: float):
        print(f"Cobrando ${monto} mediante Pasarela de Crédito (Visa/Mastercard).")

class PagoTransferencia(EstrategiaPago):
    def pagar(self, monto: float):
        print(f"Validando comprobante bancario por ${monto} de forma asíncrona.")

class ContextoPedido:
    def __init__(self, estrategia: EstrategiaPago):
        self._estrategia = estrategia

    def cambiar_metodo(self, nueva_estrategia: EstrategiaPago):
        self._estrategia = nueva_estrategia

    def ejecutar_cobro(self, monto: float):
        self._estrategia.pagar(monto)

Explicación: El objeto ContextoPedido no sabe cómo se calcula o procesa el dinero internamente. Si el usuario decide pagar con transferencia en lugar de tarjeta, simplemente cambiamos la estrategia en caliente, cumpliendo estrictamente el Principio Open/Closed (OCP).

D. Observer (Observador)

Teoría: Es la columna vertebral de las arquitecturas reactivas y los sistemas orientados a eventos (Event-Driven). Establece una relación de uno a muchos, de modo que cuando un objeto principal (Sujeto/Publisher) cambia su estado, notifica automáticamente a todos sus dependientes (Observadores/Subscribers).

# Ejemplo: Emisor de Alertas Académicas del Instituto
class SujetoNotificable:
    def __init__(self):
        self._observadores = []

    def suscribir(self, observador):
        self._observadores.append(observador)

    def notificar_todos(self, mensaje: str):
        for observador in self._observadores:
            observador.actualizar(mensaje)

class CanalWhatsApp:
    def actualizar(self, mensaje: str):
        print(f"[WhatsApp Alumno] Notificación entrante: {mensaje}")

class CapaAuditoria:
    def actualizar(self, mensaje: str):
        print(f"[Logs Sistema] Guardando en disco evento auditado: {mensaje}")

Explicación: El SujetoNotificable no conoce la implementación de WhatsApp ni de la capa de auditoría. Solo recorre su lista interna y despacha el evento, permitiendo agregar nuevos oyentes en el futuro de forma desacoplada.

💡 Sinergia Táctica de la Semana: Mientras los estilos arquitectónicos globales (como Capas o Microservicios) organizan la infraestructura macro de la empresa, los **Patrones de Diseño de la GoF** operan a nivel táctico micro, ordenando las clases internas para que sean flexibles y cumplan los principios SOLID. Puedes usar un *Strategy* dentro de la capa de dominio de una Arquitectura Limpia para resolver reglas financieras cambiantes sin alterar el software base.

7.3 Laboratorio Práctico de la Semana: Sistema de Automatización de Tareas Académicas

Problema Planteado por el Ingeniero: Se requiere implementar un sistema automatizado que procese tareas entregadas por los alumnos. El flujo obligatorio exige: 1. Validar el formato del archivo subido (.pdf), 2. Registrar la hora exacta de entrega en auditoría, y 3. Si la tarea es válida, notificar inmediatamente por correo al docente. Si un paso falla, la cadena debe detenerse inmediatamente. Solución mediante **Chain of Responsibility** y **Observer**.

# ==========================================================================
# CÓDIGO COMPLETO DEL LABORATORIO - COMPORTAMIENTO COMBINADO
# ==========================================================================

# Eslabón Base de la Cadena de Middleware Académico
class EslabonProcesamiento:
    def __init__(self, siguiente=None):
        self._siguiente = siguiente

    def procesar(self, tarea: dict) -> bool:
        if self._siguiente:
            return self._siguiente.procesar(tarea)
        return True

# Eslabón 1: Validación Estricta de Formatos
class ValidadorArchivo(EslabonProcesamiento):
    def procesar(self, tarea: dict) -> bool:
        if not tarea['archivo'].endswith('.pdf'):
            print(f" [ERROR CADENA] Archivo '{tarea['archivo']}' rechazado. Debe ser PDF.")
            return False
        print(" [OK CADENA] Formato de archivo verificado (.pdf).")
        return super().procesar(tarea)

# Eslabón 2: Registro de Tiempos y Auditoría
class RegistroAuditoria(EslabonProcesamiento):
    def procesar(self, tarea: dict) -> bool:
        import datetime
        tarea['fecha_procesado'] = datetime.datetime.now().strftime("%Y-%m-%d %H:%M:%S")
        print(f" [OK CADENA] Timestamp asentado en logs de auditoría: {tarea['fecha_procesado']}")
        return super().procesar(tarea)

# Publicador Reactivo de Eventos Académicos (Observer)
class NotificadorTareas:
    def __init__(self):
        self._suscriptores = []

    def registrar_oyente(self, suscriptor):
        self._suscriptores.append(suscriptor)

    def disparar_evento(self, tarea: dict):
        for s in self._suscriptores:
            s.notificar(tarea)

class ServicioNotificacionDocente:
    def notificar(self, tarea: dict):
        print(f"📬 [NOTIFICACIÓN DOCENTE] El estudiante entregó la tarea '{tarea['titulo']}' a tiempo.")

# --- FLUJO PRINCIPAL EN TIEMPO DE EJECUCIÓN ---
if __name__ == "__main__":
    print("=== INICIANDO PIPELINE DE EVALUACIÓN ACADÉMICA ===")
    
    # 1. Configurar los Observadores
    publicador = NotificadorTareas()
    servicio_correo = ServicioNotificacionDocente()
    publicador.registrar_oyente(servicio_correo)
    
    # 2. Configurar la Cadena de Responsabilidad (Middleware)
    pipeline = ValidadorArchivo(RegistroAuditoria())
    
    # Caso de Prueba 1: Tarea Correcta
    tarea_pablo = {"titulo": "Manual HTML Sem.7", "archivo": "manual_pablo.pdf"}
    print("\nEvaluando Tarea de Pablo:")
    if pipeline.procesar(tarea_pablo):
        publicador.disparar_evento(tarea_pablo)
        
    # Caso de Prueba 2: Tarea Incorrecta (Debe abortar la cadena)
    tarea_invalida = {"titulo": "Copia Examen", "archivo": "script.exe"}
    print("\nEvaluando Tarea Inválida:")
    if pipeline.procesar(tarea_invalida):
        publicador.disparar_evento(tarea_invalida)

Semana 8: Diseño Estratégico y APIs — Límites de Negocio y Calidad Técnica

8.1 Introducción al Diseño Estratégico

Antes de codificar o diseñar diagramas de bases de datos relacionales, los arquitectos de software deben comprender macroscópicamente cómo se divide, organiza y comunica el negocio real. El diseño estratégico no se enfoca en los detalles de clases internas, sino en definir las fronteras lógicas y asegurar que el software mapee fielmente el flujo operativo institucional.

8.2 Las Tres Aristas Clave con Ejemplos Arquitectónicos

A continuación se desarrollan los tres pilares fundamentales exigidos para modelar soluciones a gran escala:

1. Domain-Driven Design (DDD) Estratégico

Teoría: Metodología de modelado que traduce las reglas del negocio al código mediante dos conceptos críticos:

  • Lenguaje Ubicuo (Ubiquitous Language): Vocabulario común compartido estrictamente entre expertos de negocio y desarrolladores. Evita sinónimos ambiguos en las variables del código.
  • Contexto Delimitado (Bounded Context): La frontera lógica e inequívoca donde un modelo conceptual de software tiene validez absoluta.

Ejemplo en el Entorno Académico: La palabra "Matrícula" cambia radicalmente según la frontera de negocio:

  • En el Contexto Financiero, una Matrícula representa un valor económico por cobrar (Atributos: monto, estado_pago, factura_id).
  • En el Contexto Académico, representa el registro de asignaturas tomadas (Atributos: ciclo, paralelo, materias_inscritas).

Solución de Software (Código Desacoplado):

# Contexto Financiero (Maneja cobros)
class MatriculaFinanciera:
    def __init__(self, estudiante_id: str, costo_base: float):
        self.estudiante_id = estudiante_id
        self.monto_por_pagar = costo_base
        self.pagado = False

    def registrar_pago(self):
        self.pagado = True

# Contexto Académico (Maneja cupos y horarios)
class MatriculaAcademica:
    def __init__(self, estudiante_id: str, nivel: str):
        self.estudiante_id = estudiante_id
        self.nivel = nivel
        self.materias = []

    def asignar_materia(self, codigo_materia: str):
        self.materias.append(codigo_materia)

Explicación: Separar el concepto en dos clases dentro de módulos distintos evita crear un solo objeto gigante "Matrícula" lleno de if/else, permitiendo que el equipo financiero y el académico trabajen de forma independiente.

2. Ley de Conway e Inverse Conway Maneuver

Teoría: "Las organizaciones que diseñan sistemas están forzadas a producir diseños que son copias de las estructuras de comunicación de dichas organizaciones". Si los equipos de desarrollo están aislados, el software resultante tendrá APIs rotas y monolitos difíciles de acoplar.

Ejemplo Real: Si el Instituto asigna un único equipo de desarrollo mixto para hacer la app móvil, el backend y la base de datos sin dividirse las tareas, construirán un "Monolito de Espagueti".

Aplicación Práctica (Inverse Conway Maneuver): Consiste en diseñar primero la estructura del equipo humano que deseas promover (por ejemplo: Equipo de Cobros y Equipo de Notas por separado) para forzar evolutivamente a que el software se transforme de manera natural en microservicios limpios y distribuidos.

3. Deuda Técnica (El Cuadrante de Martin Fowler)

Teoría: Es el costo financiero y operativo oculto de avanzar rápido lanzando código descuidado, de mala calidad o sin documentación, sacrificando la mantenibilidad futura por una ganancia inmediata de tiempo.

Ejemplo mediante el Cuadrante de Fowler aplicado:

  • Prudente / Deliberada: "Debemos salir a producción ya para cumplir la acreditación del instituto; conocemos las fallas técnicas del código y las pagaremos con refactorización el próximo sprint".
  • Imprudente / Deliberada (Peligrosa): "No hay tiempo de diseñar APIs ni modularizar, tiren código rápido sin documentar y que luego vea el siguiente desarrollador cómo lo arregla".

8.3 Mitigación de Riesgos Mediante Contratos de API (API Contracts)

📝 Design-First: Consiste en diseñar formalmente el contrato de la API (usando herramientas como OpenAPI o Swagger) antes de escribir una sola línea de código en el backend. Esto permite que el equipo de Frontend y el de Backend avancen en paralelo utilizando datos simulados (Mocks).

🛡️ Aislamiento de Cambios: Al establecer contratos estables, se reduce drásticamente la deuda técnica inadvertida causada por actualizaciones que rompen la comunicación entre equipos de desarrollo independientes.

📌 Versionamiento Estratégico: Es obligatorio incluir esquemas de versión claros directamente en las URLs expuestas (ej: /api/v1/estudiantes) para otorgar predictibilidad total al ecosistema de integración corporativa.

8.4 Laboratorio Práctico: Contrato y Simulación de API Académica (OpenAPI Mock)

Problema del Laboratorio: El equipo frontend del IST Yavirac necesita consumir las notas de los alumnos, pero el equipo backend sigue diseñando la base de datos. Para evitar retrasos y deuda técnica, se define un contrato estricto de API y se crea un script en Python que actúa como Servidor Mock (Simulador de Contrato).

# ==========================================================================
# SIMULADOR DE CONTRATO DE API - SEMANA 8 (DESIGN-FIRST)
# ==========================================================================
import json

# Simulación de la especificación OpenAPI en formato JSON para la URL: /api/v1/estudiantes
CONTRATO_OPENAPI = """
{
    "endpoint": "/api/v1/estudiantes",
    "version": "v1.0.0",
    "metodo": "GET",
    "respuesta_esperada": {
        "status": "success",
        "data": [
            {"id": "YAV-001", "nombre": "Pablo Reyes", "promedio": 9.5},
            {"id": "YAV-002", "nombre": "Ana Belen", "promedio": 8.8}
        ]
    }
}
"""

class MockApiServer:
    def __init__(self, contrato_json: str):
        self.contrato = json.loads(contrato_json)

    def atender_peticion(self, url: str, metodo: str):
        # Validación estricta del contrato y de la versión de la URL
        if url == self.contrato["endpoint"] and metodo == self.contrato["metodo"]:
            print(f" [API CONTRACT] Petición válida para la versión: {self.contrato['version']}")
            return 200, self.contrato["respuesta_esperada"]
        else:
            print(" [API CONTRACT ERROR] URL no encontrada o contrato de versión roto.")
            return 404, {"error": "Endpoint Incompatible con el Contrato Establecido"}

# --- EJECUCIÓN DEL MOCK EMULANDO FRONTEND ---
if __name__ == "__main__":
    print("=== PROBANDO ENTORNO PARALELO FRONTEND / BACKEND ===")
    servidor_simulado = MockApiServer(CONTRATO_OPENAPI)
    
    # El Frontend consume la API v1 con total seguridad aunque el backend real no exista
    codigo_estado, respuesta = servidor_simulado.atender_peticion("/api/v1/estudiantes", "GET")
    
    print(f"Código HTTP recibido: {codigo_estado}")
    print("Datos mapeados en el Frontend:")
    print(json.dumps(respuesta, indent=4, ensure_ascii=False))
